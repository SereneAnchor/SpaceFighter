# Agent Instructions

## 目标

本项目可能较大，请优先节省 token，避免无关文件读取。

你需要遵循“先定位，再读取；先计划，再修改”的原则。不要一上来读取整个项目，也不要默认扫描所有源码。

---

## 默认行为

- 不要默认读取整个项目。
- 不要默认读取 `build/`、`out/`、`bin/`、`obj/`、`.vscode/`、`.idea/`、`assets/`、`third_party/`、`vendor/`、`external/`、`vcpkg_installed/`。
- 不要默认读取图片、音频、字体、二进制文件、编译产物、日志文件。
- 不要默认读取第三方库源码，除非我明确要求。
- 修改文件前必须先给出修改计划。
- 如果需要读取我没有指定的文件，先列出文件名和读取理由，等我确认后再读取。
- 优先读取头文件和构建文件，再读取实现文件。
- 回答尽量简洁，除非我要求详细解释。

---

## 项目分析流程

当我要求“分析项目”时，请按以下步骤执行：

1. 先查看目录结构，不打开源码文件。
2. 识别构建系统，例如 `CMakeLists.txt`、`Makefile`、`premake`、`.vcxproj`。
3. 识别入口文件，例如 `main.cpp`、`WinMain` 所在文件。
4. 识别核心目录，例如 `src/`、`include/`、`engine/`、`game/`、`core/`。
5. 列出下一步建议读取的 3-5 个文件，并说明每个文件为什么重要。
6. 等我确认后，再读取这些文件。
7. 不要一次性读取整个项目。

---

## C++ 项目优先级

优先读取：

- `README.md`
- `CMakeLists.txt`
- `Makefile`
- `.vcxproj`
- `main.cpp`
- `src/main.cpp`
- `src/` 下和问题相关的 `.cpp`
- `include/` 下和问题相关的 `.h` / `.hpp`
- 与报错或需求直接相关的类定义和实现文件

读取顺序建议：

1. 构建文件
2. 程序入口
3. 核心头文件
4. 相关实现文件
5. 问题相关文件
6. 测试文件

## 问题定位规则

当我给出 bug、报错或功能问题时，请按以下流程处理：

1. 先根据报错关键词、函数名、类名、模块名搜索相关文件。
2. 不要读取整个项目。
3. 只读取和问题直接相关的文件。
4. 如果需要更多上下文，先列出候选文件和理由。
5. 给出原因分析和修改计划。
6. 等我确认后再修改代码。

---

## 修改规则

修改前必须先输出修改计划。

修改计划中必须包含：

1. 准备修改哪些文件
2. 每个文件为什么需要修改
3. 修改的具体思路
4. 可能的风险
5. 如何验证修改是否成功

没有我的确认，不要直接修改文件。

---

## 编译错误处理规则

当我提供 C++ 编译错误时，请优先处理第一条真实错误，不要被后续连锁错误干扰。

优先查看：

- 报错文件
- 报错行附近代码
- 相关头文件
- `CMakeLists.txt`
- include 路径配置
- 宏定义
- 类型声明

不要一开始读取整个项目。

推荐流程：

1. 解释第一条错误是什么意思。
2. 定位错误涉及的文件和符号。
3. 判断是语法错误、头文件缺失、类型不完整、命名空间问题、链接问题还是构建配置问题。
4. 列出需要读取的文件。
5. 等我确认后再读取。

---

## 链接错误处理规则

当我提供链接错误时，优先检查：

- `CMakeLists.txt`
- `target_link_libraries`
- `target_sources`
- 源文件是否加入 target
- 函数声明和定义是否一致
- 类成员函数是否只声明未定义
- 静态库 / 动态库路径
- 第三方库链接配置

不要读取 `build/` 里的大量中间文件，除非我明确要求。

---

## 运行崩溃处理规则

当我提供运行时崩溃、段错误、访问冲突、空指针问题时，优先检查：

- 调用栈涉及的函数
- 对象生命周期
- 指针是否为空
- 容器迭代器是否失效
- 析构逻辑
- `new/delete` 或智能指针使用
- SDL / OpenGL / DirectX 资源释放顺序
- 多线程访问共享资源

不要直接扫描全项目。

---

## 游戏 / SDL 项目规则

如果这是 C++ 游戏项目，优先分析：

- `Game.cpp` / `Game.h`
- 主循环
- `Scene` / `MainScene`
- `Object` / `Entity`
- `Player`
- `Enemy`
- `Bullet`
- `Renderer`
- `Texture`
- `Input`
- 资源加载路径
- update / render 调用顺序

默认不要读取：

- 图片资源
- 音频资源
- 字体文件
- 大型资源目录
- 编译产物

如果问题与资源加载有关，只读取资源路径配置和加载代码，不读取资源文件本身。

---

## 输出风格

默认按以下结构回答：

1. 结论
2. 证据
3. 需要读取的文件
4. 修改计划或下一步建议

除非我要求详细解释，否则回答保持简洁。

如果我要求“只给结论”，不要展开长篇分析。

---

## 低 Token 约束

在每次任务中，请主动节省 token：

- 优先用目录结构判断项目。
- 优先用搜索定位文件。
- 优先读少量关键文件。
- 不要重复读取已经分析过的内容。
- 如果已有 `PROJECT_SUMMARY.md`，先参考它。
- 如果已有 `FILE_INDEX.md`，先根据索引选择文件。
- 如果已有 `BUG_CONTEXT.md`，先围绕其中列出的文件分析。

---

## 推荐的工作方式

当我说“分析项目”时，你应该先回复类似：

> 我将先查看目录结构和构建文件，不会读取源码。然后我会列出建议读取的 3-5 个关键文件，等待你确认。

当我说“修复问题”时，你应该先回复类似：

> 我会先定位相关文件并给出修改计划，不会直接修改。若需要额外文件，我会先说明理由并等待确认。

## 禁止行为

除非我明确要求，否则不要：

- 一次性读取整个仓库
- 读取所有 `.cpp` / `.h` 文件
- 读取 `build/`、`assets/`、`third_party/`
- 大量输出无关解释

- 自动执行危险命令
- 删除文件
- 重命名项目结构
- 改动构建系统但不说明风险

---

## 验证规则

修改后必须说明如何验证，例如：

- 运行哪条构建命令
- 运行哪个测试
- 启动哪个程序
- 检查哪个界面或日志
- 预期结果是什么

如果无法验证，请明确说明原因。
